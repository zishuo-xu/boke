# RAG 技术分享

## 一、什么是 RAG？

**RAG = Retrieval-Augmented Generation（检索增强生成）**

RAG 是一种结合检索系统和生成模型的技术，让 AI 能够基于外部知识库生成更准确、更可靠的回答。

```
用户问题 → 检索系统 → 找到相关文档 → 组装Prompt → 大模型生成 → 最终答案
```

### 为什么需要 RAG？

| 问题 | 纯生成 | RAG |
|------|--------|-----|
| 知识时效性 | 依赖训练数据 | 可实时更新知识库 |
| 幻觉问题 | 容易编造答案 | 基于真实文档回答 |
| 可追溯性 | 无法验证 | 可引用具体来源 |
| 成本 | 低 | 需要维护检索系统 |

---

## 二、RAG 工作流程

RAG 流程分为 **索引构建** 和 **检索生成** 两个阶段：

### 阶段一：索引构建（离线）

```
原始文档 → 预处理 → 切片(Chunking) → 向量化 → 存入向量数据库
```

1. **预处理**：清洗文本（去噪声、标准化格式）
2. **切片**：将长文档切成小块（chunks）
3. **向量化**：使用Embedding模型将文本转为向量
4. **存储**：将向量存入向量数据库（如 Milvus、Pinecone）

### 阶段二：检索生成（在线）

```
用户问题 → 向量化Query → 相似度检索 → 重排(Rerank) → 组装Prompt → 大模型生成
```

1. **Query 向量化**：将用户问题转为向量
2. **相似度检索**：在向量库中查找最相关的 chunks
3. **重排**：对召回结果重新排序
4. **Prompt 组装**：将相关文档和问题组合成完整 prompt
5. **生成回答**：调用大模型生成答案

## 三、关键技术与参数

### 切片策略（Chunking）

| 策略 | 原理 | 优缺点 |
|------|------|--------|
| 按字符 | 固定字符数切分 | 简单，可能切断句子 |
| 按句子 | 按句号/换行切分 | 保持语义完整，大小不均 |
| 按段落 | 按段落边界切分 | 语义连贯性强 |

**关键参数：**
- `chunk_size`：每个 chunk 的目标大小
- `overlap`：相邻 chunk 之间的重叠字符数（保持上下文连贯）

### 向量化（Embedding）

将文本映射到高维向量空间：
- 语义相似的文本 → 向量距离更近
- 向量维度：通常 768-1536 维
- 常用模型：OpenAI Embedding、Sentence-BERT、Doubao Embedding

### 相似度计算

| 方法 | 公式 | 特点 |
|------|------|------|
| 余弦相似度 | cos(θ) = (A·B) / (|A|×|B|) | 值范围 [-1, 1]，方向敏感 |
| 点积 | A·B = Σaᵢbᵢ | 值范围无界，计算效率高 |
| 欧氏距离 | d = √Σ(aᵢ-bᵢ)² | 几何直观，值越大距离越远 |

### 重排（Rerank）

向量检索的补充：
- 初次检索可能遗漏语义细节
- 重排模型能更精细地评估 Query-Doc 匹配度
- 可能会改变初始排序

## 四、切片策略对比

### 按字符切片
```
原文：深度学习是机器学习的重要技术
chunk_size=6, overlap=2

chunk_0: 深度学习是机  (位置0-5)
chunk_1: 是机器学习的  (位置4-9)
chunk_2: 学习的重要技  (位置8-13)
```

说明：每 chunk_size=6 个字符连续切分，相邻块有 overlap=2 个字符的重叠，保持上下文连贯。例如 chunk_0 末尾"是机"与 chunk_1 开头"是机"重复。

### 按句子切片
```
chunk_0: 人工智能（AI）是计算机科学的一个重要分支，致力于开发智能系统。

chunk_1: 机器学习是人工智能的一个子集，使系统能够从数据中学习。

chunk_2: 深度学习是机器学习的一个分支，使用多层神经网络分析数据。

chunk_3: 大型语言模型（LLM）是深度学习在自然语言处理领域的应用。
```

### 按段落切片
```
chunk_0: 人工智能（Artificial Intelligence，AI）是计算机科学的一个重要分支，
致力于开发能够执行通常需要人类智能的任务的系统。这包括视觉感知、语音识别、
决策制定和语言翻译等。

chunk_1: 机器学习是人工智能的一个子集，它使系统能够从数据中学习并改进性能，
而无需明确编程。深度学习则是机器学习的一个分支，使用多层神经网络来分析各种因素的数据。

chunk_2: 大型语言模型（LLM）是深度学习在自然语言处理领域的应用，能够理解
和生成人类语言。检索增强生成（RAG）技术结合了检索系统和生成模型，提高了
AI系统的准确性和可靠性。
```

## 五、每个环节详解

---

### 环节 1：原始文本

**是什么**
RAG 流程的起点，是即将进入处理流程的原始文档内容。可能来自知识库、文档库、数据库等。

**为什么**
原始文档通常包含噪声（多余空格、特殊符号）、结构不统一、格式不规范等问题，不能直接用于检索。

**怎么做**
准备高质量的文档数据：
- 确保内容完整、语义连贯
- 排除无关广告、水印等干扰信息
- 文档格式可以是 TXT、PDF、HTML、Markdown 等

**效果**
得到一份干净的原始文档，为后续处理做准备。

---

### 环节 2：预处理

**是什么**
对原始文本进行清洗和标准化的过程，包括去除噪声、统一格式、规范化文本结构。

**为什么**
- 去除多余空格、换行、特殊字符等噪声
- 统一编码格式，消除乱码隐患
- 规范化文本结构，便于后续切片
- 减少 embedding 时的干扰信息

**怎么做**
常见预处理操作：
```
1. 去除多余空白：多个空格合并为一个
2. 规范化换行：换行符统一处理
3. 清除特殊字符：移除无意义的符号
4. 统一标点：全角转半角等
5. 文本标准化：大小写统一（英文场景）
```

**效果**
得到一份干净、规范的预处理文本，字符数可能略有减少但语义保持完整。

---

### 环节 3：切片（Chunking）

**是什么**
将长文档切分成多个较小的、语义相对独立的文本块（chunks）。

**为什么**
- 大模型有上下文长度限制，无法处理完整长文档
- 完整的文档太长，难以精确匹配用户问题
- 适当的 chunk 大小能提高检索精度
- 方便定位具体的信息来源

**怎么做**
1. **选择切片策略**：
   - 按字符：固定字符数切分，简单但可能切断句子
   - 按句子：在句号、问号等处切分，保持语义完整
   - 按段落：在段落边界切分，语义连贯性强

2. **设置关键参数**：
   - `chunk_size`：每个 chunk 的目标大小（建议 300-500 字符）
   - `overlap`：相邻 chunk 之间的重叠字符数（建议 10-20% 的 chunk_size）

**效果**
得到多个语义相对完整的 chunks，每个 chunk 可以独立检索和参与计算。

---

### 环节 4：向量化（Embedding）

**是什么**
使用 Embedding 模型将文本（chunks）转换为高维向量表示。

**为什么**
- 文本的向量表示可以捕捉语义信息
- 语义相似的文本在向量空间中距离更近
- 向量运算可以高效计算文本相似度
- 为后续的相似度检索提供基础

**怎么做**
调用 Embedding API 将文本转为向量：
```
输入文本 → Embedding 模型 → 输出向量

示例（简化表示）：
"人工智能是计算机科学" → [0.12, -0.34, 0.56, ..., 0.89] (1024维)
```

常用 Embedding 模型：
- OpenAI Embedding (1536维)
- Sentence-BERT (768维)
- Doubao Embedding (支持多语言)

**效果**
每个 chunk 被映射为一个高维向量，向量维度通常 768-1536 维。语义相似的文本，向量距离更近。

---

### 环节 5：用户提问（Query 输入）

**是什么**
用户提出的问题或查询意图，是 RAG 检索的触发点。

**为什么**
- Query 决定了检索的方向和目标
- Query 本身也需要向量化才能参与相似度计算
- Query 与 chunks 在同一向量空间中进行匹配
- Query 的质量直接影响检索效果

**怎么做**
1. 用户输入自然语言问题
2. 系统对 Query 进行预处理（与文档预处理类似）
3. 将预处理后的 Query 发送给 Embedding 模型
4. 得到 Query 对应的向量表示

**效果**
得到 Query 的向量表示，准备进入检索流程。

---

### 环节 6：相似度检索

**是什么**
在向量数据库中查找与 Query 向量最相似的 chunks，使用余弦相似度等度量方法。

**为什么**
- 向量检索能捕捉语义相似性，不只是关键词匹配
- 高效处理大规模向量数据
- 可以快速召回 Top-K 个最相关结果

**怎么做**
1. 计算 Query 向量与所有 chunk 向量的相似度
2. 常用相似度度量：
   - **余弦相似度**：cos(θ) = (A·B) / (|A|×|B|)，值范围 [-1, 1]
   - **点积**：A·B = Σaᵢbᵢ，值越大越相似
   - **欧氏距离**：d = √Σ(aᵢ-bᵢ)²，值越小越相似
3. 按相似度降序排列，召回 Top-K 个结果

```
Query 向量: [0.1, 0.2, 0.3, ...]
     ↓
与所有 chunks 计算相似度
     ↓
chunk_2: 0.95 ✓ (最相关)
chunk_0: 0.87 ✓
chunk_3: 0.72
chunk_1: 0.45
```

**效果**
召回 Top-K 个最相关的 chunks，并附带相似度分数。Top-K 外的 chunks 被标记为"未命中"。

---

### 环节 7：重排（Rerank）

**是什么**
对初步召回的结果进行更精细的排序，使用重排模型重新评估 Query 与 Document 的匹配度。

**为什么**
- 向量检索可能遗漏语义细节
- 部分 chunk 与问题表面相似但实际不相关
- 重排模型能更精细地理解 Query 和 Document 的匹配度
- 可以显著提升最终答案的质量

**怎么做**
1. 使用重排模型（如 BGE-Reranker、Cohere Rerank）对 Top-K 结果重新评分
2. 重排模型考虑更多语义因素
3. 按新分数重新排序

```
初排: chunk_2 (0.95) > chunk_0 (0.87) > chunk_1 (0.72)
重排: chunk_0 (0.92) > chunk_2 (0.89) > chunk_1 (0.65)
     (chunk_0 与问题意图更匹配，上升至第一位)
```

**效果**
得到更精确的排序结果，可能改变初始检索的顺序，提高答案质量。

---

### 环节 8：Prompt 组装

**是什么**
将系统指令、检索到的相关文档（Context）和用户问题组合成完整的 Prompt，发送给大模型。

**为什么**
- 大模型需要明确的上下文才能准确回答
- Context 帮助模型理解问题背景
- 系统指令设定模型的角色和回答规则
- 组装方式直接影响回答质量

**怎么做**
典型 Prompt 结构：
```
【系统指令】
你是一个专业的AI助手。你的任务是基于提供的上下文来回答用户问题。
如果上下文中没有相关信息，请如实告知。

【上下文】
chunk_1: 人工智能（AI）是计算机科学的一个重要分支...
chunk_2: 机器学习是人工智能的一个子集...

【用户问题】
什么是机器学习？

【回答要求】
请基于上述上下文回答问题。
```

组装原则：
- Context 要精选，不是越多越好
- 明确指示模型基于上下文回答
- 如果上下文不足，让模型承认不知道

**效果**
得到一个完整的 Prompt，可以发送给大模型生成最终答案。

---

### 环节 9：答案生成

**是什么**
大模型基于组装好的 Prompt 生成最终的回答。

**为什么**
- 模型综合上下文信息和问题生成答案
- 答案基于真实检索结果，更可靠
- 可以追溯具体的信息来源
- 比纯生成更不易产生幻觉

**怎么做**
1. 将组装好的 Prompt 发送给大模型（如 GPT-4、Claude、文心一言等）
2. 模型根据上下文生成回答
3. 返回最终答案给用户

```
Prompt → 大模型 → 最终答案

示例：
问题：什么是机器学习？
答案：机器学习是人工智能的一个子集，它使系统能够从数据中学习
并改进性能，而无需明确编程。这项技术通过算法识别数据模式，
并利用这些模式进行预测和决策。
```

**效果**
得到基于真实文档生成的回答，答案可追溯、质量更高。

## 六、最佳实践建议

### 切片优化
- Chunk 大小通常 300-500 字符
- Overlap 建议 10-20% 的 chunk_size
- 保持语义完整，优先按句子/段落切分

### 检索优化
- Top K 不宜过大（通常 3-10）
- 根据实际效果调整重排策略
- 可以添加元数据过滤（如时间、类别）

### Prompt 工程
- Context 要精选，不是越多越好
- 明确指示模型基于上下文回答
- 必要时让模型承认不知道

### Embedding 模型选择
- 中文场景优先选择中文效果好的模型
- 根据文档类型选择合适的 embedding 服务
- 注意向量维度对存储和检索性能的影响

## 七、常见问题

**Q: RAG 和微调有什么区别？**
A: 微调是让模型适应特定任务/风格；RAG 是给模型提供外部知识。RAG 更适合知识频繁更新的场景。

**Q: 什么时候不需要 RAG？**
A: 如果任务不依赖外部知识、答案可以从模型知识中准确回答、或者实时性要求不高，可以不用 RAG。

**Q: 如何评估 RAG 效果？**
A: 可以从召回率（相关文档是否被召回）、答案准确率（答案是否基于正确文档）、响应时间等维度评估。
