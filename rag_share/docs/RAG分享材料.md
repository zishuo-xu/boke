# RAG 技术分享

## RAG 概述

### 什么是 RAG？

RAG = Retrieval-Augmented Generation（检索增强生成）

RAG 是一种结合检索系统和生成模型的技术，让 AI 能够基于外部知识库生成更准确、更可靠的回答。

```
用户问题 → 检索系统 → 找到相关文档 → 组装Prompt → 大模型生成 → 最终答案
```

### 为什么需要 RAG？

| 问题 | 纯生成 | RAG |
|------|--------|-----|
| 知识时效性 | 依赖训练数据 | 可实时更新知识库 |
| 幻觉问题 | 容易编造答案 | 基于真实文档回答 |
| 可追溯性 | 无法验证 | 可引用具体来源 |
| 成本 | 低 | 需要维护检索系统 |

### RAG 工作流程

RAG 流程分为两个阶段：

**索引构建（离线）**
```
原始文档 → 预处理 → 切片 → 向量化 → 存入向量数据库
```

**检索生成（在线）**
```
用户问题 → 向量化Query → 相似度检索 → 重排 → 组装Prompt → 大模型生成
```

## 各步骤详解

---

### 步骤 1：原始文本

**是什么**

RAG 流程的起点，指即将进入处理流程的原始文档。文档可能来自知识库、文档库、数据库等。

**为什么**

原始文档通常存在以下问题，不适合直接检索：
- 包含噪声：多余空格、特殊符号、HTML 标签等
- 结构不统一：格式杂乱，缺乏规范性
- 包含干扰信息：如广告、水印、脚注等

**怎么做**

准备高质量的文档数据：
- 确保内容完整、语义连贯
- 排除广告、水印等干扰信息
- 支持 TXT、PDF、HTML、Markdown 等多种格式

**效果**

得到一份原始文档，为后续处理做准备。

---

### 步骤 2：预处理

**是什么**

对原始文本进行清洗和标准化的过程，包括去除噪声、统一格式、规范文本结构。

**为什么**

预处理可以解决以下问题：
- 去除多余空格、换行、特殊字符等噪声
- 统一编码格式，避免乱码
- 规范化文本结构，便于后续切片
- 减少 embedding 时的干扰信息

**怎么做**

常见预处理操作：
```
1. 去除多余空白：多个空格合并为一个
2. 规范化换行：统一换行符处理
3. 清除特殊字符：移除无意义的符号
4. 统一标点：全角转半角等
```

**效果**

得到一份干净、规范的预处理文本，字符数可能略有减少但语义保持完整。

---

### 步骤 3：切片（Chunking）

**是什么**

将长文档切分成多个较小的、语义相对独立的文本块（chunks）。

```
原始文档（10000字符）
        ↓
    切片处理
        ↓
chunk_0: "人工智能（AI）是计算机科学的一个分支..."
chunk_1: "机器学习是人工智能的一个子集..."
chunk_2: "深度学习是机器学习的一个重要分支..."
chunk_3: "大型语言模型（LLM）是..."
...
```

**为什么**

切片是 RAG 中的关键步骤，原因如下：

1. **模型上下文限制**：大模型有 token 数量限制（如 4K、8K、128K），无法一次性处理完整长文档

2. **提高检索精度**：文档太长会导致关键信息被稀释，切成小块后更容易精确匹配用户问题

3. **来源可追溯**：明确知道答案来自哪个 chunk，而非整篇文档

4. **成本控制**：发送给模型的 context 更少，响应更快、成本更低

**怎么做**

**选择切片策略：**

| 策略 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| 按字符 | 固定字符数切分 | 简单、大小均匀 | 可能切断句子 |
| 按句子 | 按句号、问号等切分 | 语义完整 | 大小不均匀 |
| 按段落 | 按段落边界切分 | 语义连贯 | 可能包含噪声 |
| 递归切片 | 按层级尝试多种方式 | 平衡效果 | 复杂度高 |

**按字符切片示例：**
```
原文：深度学习是机器学习的重要技术
chunk_size=6, overlap=2

chunk_0: 深度学习是机  (位置0-5)
chunk_1: 是机器学习的  (位置4-9)  ← 与上一块重叠2字符
chunk_2: 学习的重要技  (位置8-13)
```

**按句子切片示例：**
```
原文：
"人工智能（AI）是计算机科学的一个重要分支，致力于开发智能系统。机器学习是人工智能的一个子集。深度学习是机器学习的一个重要分支。"

按句子切分后：
chunk_0: 人工智能（AI）是计算机科学的一个重要分支，致力于开发智能系统。
chunk_1: 机器学习是人工智能的一个子集。
chunk_2: 深度学习是机器学习的一个重要分支。
```

**按段落切片示例：**
```
原文：
【段落1】人工智能（Artificial Intelligence，AI）是计算机科学的一个重要分支，致力于开发能够执行通常需要人类智能的任务的系统。

【段落2】机器学习是人工智能的一个子集，它使系统能够从数据中学习并改进性能。深度学习则是机器学习的一个分支，使用多层神经网络。

【段落3】大型语言模型（LLM）是深度学习在自然语言处理领域的应用，能够理解和生成人类语言。

按段落切分后：
chunk_0: 人工智能（Artificial Intelligence，AI）是计算机科学的一个重要分支，致力于开发能够执行通常需要人类智能的任务的系统。
chunk_1: 机器学习是人工智能的一个子集，它使系统能够从数据中学习并改进性能。深度学习则是机器学习的一个分支，使用多层神经网络。
chunk_2: 大型语言模型（LLM）是深度学习在自然语言处理领域的应用，能够理解和生成人类语言。
```

**关键参数设置：**

| 参数 | 说明 | 建议值 | 注意事项 |
|------|------|--------|----------|
| chunk_size | 每个 chunk 的目标大小 | 300-500 字符 | 太小丢失上下文，太大降低精度 |
| overlap | 相邻 chunk 重叠字符数 | 10-20% chunk_size | 太多增加冗余，太少丢失边界信息 |

**效果**

得到多个语义相对完整的 chunks，每个 chunk 可以独立检索。切片的质量直接影响后续检索和回答的效果。

### 步骤 4：向量化（Embedding）

**是什么**

将文本转换为数值向量的过程。在向量空间中，每个文本被表示为一个由数字组成的有序列表。

```
文本 "人工智能" → 向量 [0.12, -0.34, 0.56, 0.23, ..., -0.78]
文本 "机器学习" → 向量 [0.45, -0.12, 0.67, 0.89, ..., 0.34]
文本 "做菜方法" → 向量 [-0.23, 0.56, -0.34, 0.12, ..., 0.67]
```

向量中的每个数字称为"维度"，一个向量可以有几百到几千个维度。

**为什么**

1. **计算机只能处理数字**：文本语义无法被计算机直接理解，需要转换为数值形式

2. **捕捉语义关系**：语义相似的文本在向量空间中距离更近

   ```
   示例（简化二维空间）：
                    ↑ "机器学习" (0.8, 0.6)
                    |
                    |  "深度学习" (0.7, 0.5)
                    |
   文本向量空间 ----|------------------------→
                    |  "人工智能" (0.5, 0.3)
                    |
                    |  "做菜方法" (-0.3, -0.5)
                    |
   ```

3. **支持数学运算**：可以在向量空间中进行相似度计算

**怎么做**

1. **选择 Embedding 模型**：

   | 模型 | 维度 | 特点 |
   |------|------|------|
   | OpenAI text-embedding-3-large | 3072 | 效果好，支持多语言 |
   | OpenAI text-embedding-3-small | 1536 | 效果好，轻量级 |
   | Sentence-BERT | 768 | 适合句子匹配 |
   | Doubao Embedding | 1024 | 中文效果好 |

2. **调用 API 获取向量**：

   ```python
   # 示例：调用 Doubao Embedding API
   response = requests.post(
       "https://ark.cn-beijing.volces.com/api/v3/embeddings",
       headers={"Authorization": f"Bearer {API_KEY}"},
       json={
           "model": "doubao-embedding-vision-251215",
           "input": [{"type": "text", "text": "机器学习是人工智能的重要分支"}]
       }
   )
   embedding = response.json()["data"]["embedding"]
   # 输出：[0.12, -0.34, 0.56, ..., 0.89] (1024维)
   ```

3. **向量存储**：存入向量数据库

   ```
   Chunk ID    | 向量
   ------------|----------------------------------
   chunk_0    | [0.12, -0.34, 0.56, ..., 0.89]
   chunk_1    | [0.45, -0.12, 0.67, ..., 0.34]
   chunk_2    | [-0.23, 0.56, -0.34, ..., 0.67]
   ```

### 步骤 5：用户提问

**是什么**

用户提出的问题，是 RAG 检索的触发点。

**为什么**

- Query 决定了检索的方向和目标
- Query 需要向量化才能与 chunks 进行相似度计算
- Query 与 chunks 在同一向量空间中进行匹配
- Query 的质量直接影响检索效果

**怎么做**

1. 用户输入自然语言问题
2. 对 Query 进行预处理（与文档预处理类似）
3. 将预处理后的 Query 发送给 Embedding 模型
4. 得到 Query 的向量表示

### 步骤 6：相似度检索

---

#### 1. 什么是相似度检索

根据语义相似度在文档库中找到与用户问题最相关的内容。

核心思想：将文本转换为向量，在向量空间中找距离最近的邻居。

```
用户问题 "如何学习机器学习？"
         ↓
     向量化 Query
         ↓
    [0.12, -0.34, 0.56, ...]  (1024维向量)
         ↓
    在向量空间中找最近邻
         ↓
返回语义最相关的 chunks
```

**相似度度量方式：**

| 方法 | 公式 | 特点 |
|------|------|------|
| 余弦相似度 | cos(θ) = (A·B)/(\|A\|×\|B\|) | 衡量方向，不受长度影响 |
| 点积 | A·B = Σaᵢbᵢ | 简单高效，对长度敏感 |
| 欧氏距离 | d = √Σ(aᵢ-bᵢ)² | 几何距离，越小越相似 |

---

#### 2. 为什么需要混合检索

**纯向量检索的局限**

向量检索通过语义理解匹配，但存在以下问题：

```
用户问题："苹果最新手机"

向量检索结果：
1. "iPhone 15 Pro" (0.92)        ← 语义相关但缺少关键词
2. "手机处理器对比" (0.85)
3. "Apple发布会" (0.80)

问题：专有名词"苹果"（公司名）容易被语义相近的通用描述稀释
```

**混合检索 = 向量检索 + 关键词检索**

| | 纯向量检索 | 混合检索 |
|---|---|---|
| 匹配方式 | 语义相似度 | 语义 + 关键词双重匹配 |
| 专有名词 | 可能被稀释 | 关键词精准命中 |
| 同义词 | 理解能力强 | 同样支持 |

**业界主流融合算法：RRF**

**RRF (Reciprocal Rank Fusion，倒数排名融合)** 是一种简单且高效的算法，用于将来自不同搜索算法的结果列表（例如：向量检索和全文检索）合并成一个统一的排名列表。

在混合检索（Hybrid Search）中，它解决了如何公平比较“语义距离分数”和“文本匹配分数”的难题。

---

### 1. 核心原理：只看排名，不看分数
向量搜索输出的是**距离（如 0.123）**，全文搜索输出的是**相关性得分（如 15.8）**。这两者就像“公里”和“公斤”一样无法直接相加。

**RRF 的做法是：** 抛弃原始分数，只记录文档在各自列表中的**排名次序**。



### 2. 数学公式
对于每一个出现在结果集中的文档 $d$，其 RRF 分数计算如下：

$$RRFscore(d) = \sum_{r \in R} \frac{1}{k + r(d)}$$

* **$R$**：所有的检索结果列表（如：向量列表、关键词列表）。
* **$r(d)$**：文档 $d$ 在当前列表中的**排名位置**（从 1 开始）。
* **$k$**：一个常数（通常取 **60**），用于平滑排名，防止排名靠后的文档权重波动过大。

---

### 3. 举个例子
假设我们要搜索“PostgreSQL 插件”，有两路召回结果：

| 文档 ID | 向量检索排名 | 全文检索排名 |
| :--- | :--- | :--- |
| **文档 A** | 1 | 2 |
| **文档 B** | 2 | 1 |
| **文档 C** | 3 | 未上榜 (假设排 100+) |

**计算文档 A 的得分 (设 $k=60$)：**
$$Score(A) = \frac{1}{60+1} + \frac{1}{60+2} \approx 0.01639 + 0.01612 = 0.03251$$

**计算文档 C 的得分：**
$$Score(C) = \frac{1}{60+3} + 0 \approx 0.01587$$

**结论：** 文档 A 和 B 在两路中表现都很靠前，最终得分最高；文档 C 虽然在语义上还行，但缺乏关键词支撑，排名会掉下来。

---

### 4. 为什么 RRF 备受推崇？
1.  **无需调优**：不像“加权平均”需要你拍脑袋决定向量占 70% 还是文本占 30%，RRF 默认就能跑出极好的效果。
2.  **“双路通过”原则**：它偏爱那些在多个维度都能排进前列的结果，这通常意味着结果更可靠。
3.  **对离群值不敏感**：如果某个算法给出了一个极高的异常分数，RRF 只看它的排名（第一名就是 1），不会被异常分数带偏。
4.  **实现简单**：在 PostgreSQL 中只需要几个 `CTE` 和 `RANK()` 函数就能实现。

---

### 5. RRF 的局限性
虽然 RRF 很好用，但它毕竟只是“粗排”。它无法感知文档之间微小的质量差异。例如，第一名比第二名好出 10 倍，还是只精准了一点点？RRF 无法区分。

这就是为什么在高级系统中，通常先用 **RRF 做混合召回**（选出前 50 个），再用 **Rerank（重排模型）** 进行最终精排的原因。

你是打算在 PostgreSQL 的 SQL 逻辑中直接实现这个算法吗？

#### 3. 为什么需要向量数据库

**普通数据库能做什么**

```sql
SELECT * FROM docs WHERE content LIKE '%苹果%'
```
只能精确匹配关键词，无法理解语义。

**向量数据库的核心能力**

| 能力 | 说明 | 普通数据库 | 向量数据库 |
|------|------|-----------|-----------|
| 向量存储 | 高维向量高效存储 | ✗ | ✓ |
| ANN 索引 | 近似最近邻快速检索 | ✗ | ✓ |
| 混合检索 | 向量 + 关键词同时检索 | ✗ | ✓ |
| 相似度计算 | 余弦/点积/欧氏距离 | ✗ | ✓ |
| 海量数据 | 亿级向量毫秒查询 | ✗ | ✓ |

**ANN 算法（近似最近邻）**

暴力遍历所有向量在海量数据下不现实，需要 ANN 算法加速：

| 算法 | 原理 | 速度 | 精度 |
|------|------|------|------|
| HNSW | 构建分层图索引 | 快 | 很高 |
| IVF | 先聚类再在相关簇中搜索 | 中 | 高 |
| PQ | 向量量化压缩 | 最快 | 中 |

**向量数据库查询示例**

```sql
-- 普通数据库
SELECT * FROM docs WHERE content = '苹果手机'

-- 向量数据库
SELECT * FROM docs
ORDER BY vector_distance(embedding, query_embedding)
LIMIT 10
```

**常见的向量数据库**

| 数据库 | 特点 | 适用场景 |
|--------|------|----------|
| Milvus | 开源、可私有部署 | 大规模生产环境 |
| Qdrant | 高性能、支持混合检索 | 需要精确排序 |
| Chroma | 轻量级、易上手 | 快速验证 |
| PostgreSQL + pgvector | 关系型数据库扩展 | 已有 PG 基础设施 |
| Elasticsearch | 内置向量检索 | 已有 ES 基础设施 |

**何时使用混合检索**

| 场景 | 推荐方式 | 原因 |
|------|----------|------|
| 通用问答 | 纯向量检索 | 语义理解足够 |
| 专有名词/型号/代码 | 混合检索 | 关键词精准命中 |
| 商品搜索 | 混合检索 | 既要语义又要关键词 |


### 步骤 7：重排（Rerank）

**是什么**

在向量检索召回 Top-K 结果后，使用更精细的重排模型对结果进行二次排序。

向量检索只能捕捉表层语义相似度，而重排模型能更深入理解 Query 和 Document 之间的语义关系，重新计算相关性分数并排序。

```
检索流程：
Query → 向量检索（快速筛选）→ Top-K 结果 → 重排模型（精细排序）→ 最终排序结果
```

**为什么**

1. **向量检索存在局限**：
   - 只能捕捉文本的表层语义相似度
   - 无法理解深层的查询意图
   - 可能召回"字面相似但语义不相关"的文档

2. **表面相似 ≠ 真正相关**：

   ```
   用户问题："如何用 Python 写一个网页爬虫？"

   向量检索召回：
   chunk_A: "Python 是一种编程语言，广泛用于数据分析" (相似度 0.92)
   chunk_B: "爬虫是一种自动化获取网页数据的技术" (相似度 0.88)
   chunk_C: "JavaScript 常用于网页前端开发" (相似度 0.75)

   问题：chunk_A 字面包含 Python，但实际不回答用户问题
   ```

3. **重排模型的优势**：
   - 使用更大的模型（如 Cross-Encoder）进行精细匹配
   - 能理解查询意图和文档内容的深层关系
   - 综合考虑更多语义因素

**怎么做**

1. **两阶段检索架构**：

   ```
   第一阶段 - 向量检索（召回）：
   - 使用 ANN 算法快速查找候选结果
   - 召回 Top-50 或 Top-100 个候选结果
   - 优点：速度快、支持海量数据

   第二阶段 - 重排（精排）：
   - 使用 Cross-Encoder 模型精细打分
   - 从候选结果中选出 Top-5 或 Top-10
   - 优点：精度高、效果好
   ```

2. **常用重排模型**：

   | 模型 | 类型 | 特点 |
   |------|------|------|
   | BGE-Reranker | 开源 | 效果好，支持中文 |
   | Cohere Rerank | 商业 API | 效果好，易于集成 |
   | Jina Reranker | 商业/开源 | 多语言支持 |
   | BM25 + 规则 | 传统 | 可作为补充策略 |

3. **重排示例**：

   ```
   用户问题：什么是机器学习？

   向量检索初排结果：
   chunk_2: "机器学习是人工智能的一个分支" (0.95)
   chunk_0: "人工智能技术发展迅速" (0.87)
   chunk_3: "深度学习是机器学习的一个分支" (0.82)
   chunk_1: "编程语言 Python 入门教程" (0.78)

   重排模型重新评估后：
   chunk_2: "机器学习是人工智能的一个分支" (0.95 → 0.98)  ✓ 最相关
   chunk_3: "深度学习是机器学习的一个分支" (0.82 → 0.91)  ↑ 上升

   ```

4. **何时需要重排**：
   - 检索效果不佳，Top-K 结果不够精准
   - 需要更精确的排序来提升答案质量
   - 对回答质量要求较高的场景
   - 注意：重排会增加延迟和成本，简单场景可跳过

### 步骤 8：Prompt 组装

**是什么**

将系统指令、检索到的相关文档（Context）和用户问题组合成完整的 Prompt，发送给大模型。

**为什么**

- 大模型需要明确的上下文才能准确回答
- Context 帮助模型理解问题背景
- 系统指令设定模型的角色和回答规则
- 组装方式直接影响回答质量

**怎么做**

典型 Prompt 结构：
```
【系统指令】
你是一个专业的AI助手。你的任务是基于提供的上下文来回答用户问题。
如果上下文中没有相关信息，请如实告知。

【上下文】
chunk_1: 人工智能（AI）是计算机科学的一个重要分支...
chunk_2: 机器学习是人工智能的一个子集...

【用户问题】
什么是机器学习？

【回答要求】
请基于上述上下文回答问题。
```

组装原则：
- Context 要精选，不是越多越好
- 明确指示模型基于上下文回答
- 如果上下文不足，让模型承认不知道

**效果**

得到一个完整的 Prompt，可以发送给大模型生成最终答案。

---

### 步骤 9：答案生成

**是什么**

大模型基于组装好的 Prompt 生成最终的回答。

**为什么**

- 模型综合上下文信息和问题生成答案
- 答案基于真实检索结果，更可靠
- 可以追溯具体的信息来源
- 比纯生成更不易产生幻觉

**怎么做**

1. 将组装好的 Prompt 发送给大模型（如 GPT-4、Claude、文心一言等）
2. 模型根据上下文生成回答
3. 返回最终答案给用户

```
示例：
问题：什么是机器学习？
答案：机器学习是人工智能的一个子集，它使系统能够从数据中学习
并改进性能，而无需明确编程。这项技术通过算法识别数据模式，
并利用这些模式进行预测和决策。
```

**效果**

得到基于真实文档生成的回答，答案可追溯、质量更高。

## 总结

RAG 的核心价值：

1. **知识时效性**：可实时更新知识库，无需重新训练模型
2. **减少幻觉**：答案基于真实文档，更可靠
3. **可追溯**：可以引用具体的信息来源
4. **成本效率**：比微调更低，适用于频繁更新的知识场景
