# 极简自然语言饮食热量&营养追踪器 需求文档（V1.0 核心功能版）
**文档版本**：V1.0
**核心定位**：个人自用轻量化Web应用，主打减脂/增肌场景的饮食追踪，极致简化记录流程，AI赋能营养分析，核心数据全量存储于服务器实现永久持久化
**核心原则**：优先保证核心功能落地，砍掉所有非核心需求，极简开发、快速上线
**硬性约束**：V1.0不使用MySQL等需独立部署的重型数据库，仅采用支持标准SQL语法的轻量单文件数据库，后端采用Python技术栈，LLM模型可灵活配置

---

## 一、项目概述
### 1.1 项目背景
市面上主流饮食记录APP普遍存在广告多、操作繁琐、学习成本高的问题，记录一顿饭需手动搜索食物、精准输入重量，普通人难以长期坚持；浏览器本地存储方案，存在换设备、清缓存导致数据丢失的风险。本项目为个人自用工具，在保留极简操作、AI赋能核心优势的基础上，将核心数据全量存储于自有服务器，解决数据丢失问题，同时做到轻量化、易开发、快速落地。

### 1.2 核心目标
1.  **极致简化记录流程**：通过自然语言一句话完成饮食记录，无需手动选食物、输重量，降低记录门槛
2.  **服务器级数据持久化**：用户所有核心数据全量存储于自有服务器，彻底解决浏览器清缓存、换设备导致的数据丢失问题，支持跨设备数据同步
3.  **轻量SQL数据库适配**：V1.0采用轻量单文件SQL数据库，无需独立部署数据库服务，支持标准SQL语法，零运维成本
4.  **精准营养追踪能力**：自动完成热量+三大营养素（蛋白质/碳水/脂肪）计算，实时对比个人目标，直观展示摄入进度
5.  **个性化AI饮食指导**：基于个人目标和摄入数据，给出可落地的饮食优化建议，贴合减脂/增肌核心需求

### 1.3 版本约束
| 约束项 | 明确要求 |
|--------|----------|
| 数据库 | V1.0禁用MySQL、PostgreSQL等需独立部署的重型数据库，仅使用支持标准SQL的轻量单文件数据库 |
| 运维要求 | 数据库无需单独配置、启动、维护，后端服务启动即可自动完成数据库初始化、表结构创建 |
| 功能范围 | 仅开发个人自用核心功能，不做多用户、权限分级、商业化相关的冗余功能 |
| 扩展性 | 数据库设计、业务逻辑与数据库类型解耦，后续切换至商业数据库仅需修改连接配置，无需改动核心业务代码 |

## 二、用户角色
本项目为个人自用工具，仅设置单一用户角色，无复杂权限体系：
- **角色名称**：个人使用者
- **权限范围**：全功能无限制使用权限，包括个人信息配置、饮食记录全量操作、LLM模型配置、AI分析、数据导出等所有功能

## 三、核心功能需求
### 模块一：个人基础信息与营养目标管理模块
**核心定位**：为营养计算、目标对比提供基础依据，数据全量同步至服务器存储
| 功能点 | 详细需求描述 |
|--------|--------------|
| 基础身体信息录入 | 支持填写/修改核心信息：性别、年龄、身高(cm)、体重(kg)、体脂率（选填）、日常活动水平（久坐/轻度活动/中度活动/重度活动/极重度活动） |
| 身材目标选择 | 支持3种核心目标单选：减脂/增肌/维持体重，选中后自动匹配对应的热量、宏量营养素计算规则 |
| 营养目标自动生成 | 1. 基于用户基础信息，自动计算基础代谢BMR、每日总热量消耗TDEE<br>2. 根据目标自动生成每日热量目标：减脂=TDEE-300~500kcal、增肌=TDEE+200~500kcal、维持=TDEE<br>3. 自动生成三大营养素每日目标，匹配减脂/增肌场景通用标准 |
| 目标自定义修改 | 支持用户手动修改自动生成的热量目标、三大营养素目标 |
| 数据持久化 | 用户填写的所有信息、配置的营养目标，实时同步存储至服务器数据库，页面刷新、换设备后自动拉取最新配置，无需重复填写 |

### 模块二：自然语言饮食记录核心模块
**核心定位**：产品核心竞争力，一句话完成饮食录入，所有记录全量存储于服务器
| 功能点 | 详细需求描述 |
|--------|--------------|
| 自然语言输入 | 首页核心位置设置大尺寸多行输入框，支持用户输入任意自然语言饮食描述，支持按餐次标注（早餐/午餐/晚餐/加餐），也可无标注自动识别 |
| 快速提交触发 | 支持回车、点击「记录并分析」按钮两种提交方式，核心操作步骤不超过2步 |
| AI结构化解析 | 提交后，自动通过LLM完成解析：识别食物名称、烹饪方式、食用份量，对模糊份量进行标准化重量估算，精准计算每一种食物的热量、蛋白质、碳水、脂肪数值，以及本次记录的汇总数据 |
| 识别结果展示与编辑 | 1. 前端清晰展示解析后的结构化结果，包括食物明细列表、本次记录汇总数据<br>2. 支持用户手动修改识别结果，包括食物名称、重量、营养数值，支持单条食物删除、新增自定义食物，修改后自动重新计算汇总数据 |
| 记录保存与持久化 | 用户确认后，记录实时同步存储至服务器数据库，同步更新当日营养摄入看板；换设备、刷新页面后，可随时拉取历史所有记录 |
| 历史记录管理 | 1. 支持按日期查询、查看任意历史日期的全量饮食记录<br>2. 支持单条饮食记录的编辑、删除，操作后实时同步至服务器<br>3. 支持历史常吃记录一键复用，无需重复输入 |

### 模块三：LLM模型配置与AI分析模块
**核心定位**：产品核心能力层，支持LLM模型灵活配置，完成自然语言解析、饮食建议生成
| 功能点 | 详细需求描述 |
|--------|--------------|
| LLM模型灵活配置 | 1. 支持对接主流LLM模型，包括Anthropic Claude、OpenAI GPT等，支持在页面配置模型类型、API密钥、模型版本、接口地址<br>2. 配置信息实时存储至服务器数据库，换设备后无需重复配置 |
| 提示词模板配置 | 内置标准化的「食物识别营养计算」「饮食建议生成」提示词模板，支持用户在页面自定义修改、保存模板，修改后的模板持久化存储至服务器 |
| 当日实时饮食建议 | 每新增一条饮食记录后，自动触发AI分析，基于当日摄入数据、个人营养目标，生成实时优化建议，包括营养素缺口/超标提醒、食材替换建议、下一餐饮食规划 |
| 周度饮食复盘建议 | 支持一键生成近7天饮食复盘报告，包括本周摄入趋势、营养素达标情况、饮食结构问题分析、可落地的优化建议 |
| 自定义饮食问答 | 内置自由提问框，支持用户主动发起个性化饮食相关提问，LLM给出针对性解答 |
| 异常兜底处理 | LLM接口调用超时、失败时，前端给出友好提示，支持用户手动录入饮食数据，保障基础记录功能正常使用 |

### 模块四：每日营养目标追踪看板模块
**核心定位**：数据可视化核心，用户每日高频查看的核心页面
| 功能点 | 详细需求描述 |
|--------|--------------|
| 当日核心数据概览 | 首页顶部核心位置展示：当日总摄入热量、目标热量、剩余可摄入/超量热量，用进度条可视化展示完成比例 |
| 三大营养素进度追踪 | 分别展示蛋白质、碳水、脂肪的「已摄入/目标值」，搭配独立进度条、占比饼图，直观展示每一项的达标情况 |
| 分餐次统计 | 按早餐/午餐/晚餐/加餐分类，展示每一餐的热量、营养素摄入占比，清晰查看饮食分布 |
| 历史数据查询 | 1. 内置日历组件，支持选择任意历史日期，查看对应日期的饮食记录、营养统计数据<br>2. 支持左右滑动快速切换前一天/后一天的记录 |
| 适配要求 | 所有进度条、图表轻量化实现，适配PC/移动端，数据实时从服务器拉取 |

### 模块五：服务器数据持久化管理模块
**核心定位**：本版本核心模块，保障数据永久存储、跨设备同步
| 功能点 | 详细需求描述 |
|--------|--------------|
| 全量数据自动同步 | 用户所有操作产生的数据，包括个人信息、饮食记录、LLM配置、提示词模板，均实时自动同步存储至服务器数据库，无需用户手动操作 |
| 数据拉取规则 | 页面初始化时，自动从服务器拉取用户全量基础配置、当日饮食记录；历史数据按日期按需拉取，提升页面加载速度 |
| 数据导出功能 | 支持用户将全量饮食记录、营养统计数据，导出为Excel/CSV格式文件 |
| 基础备份功能 | 支持用户一键生成全量数据备份文件，下载至本地存储 |

## 四、服务器端数据持久化核心需求
### 4.1 数据库选型要求
1.  V1.0必须采用**轻量单文件型SQL数据库**，无需独立安装、部署、启动数据库服务，无需额外运维操作
2.  完全支持标准SQL语法，包括`SELECT/INSERT/UPDATE/DELETE`等常用数据操作语句
3.  后端服务启动时，自动完成数据库文件初始化、表结构自动创建，无需手动执行SQL脚本建表
4.  数据库设计与业务逻辑解耦，预留后续无缝升级至商业数据库的能力

### 4.2 核心数据存储范围
所有用户核心数据必须全量存储于服务器数据库，具体包括：
1.  用户基础信息与营养目标数据：身体信息、身材目标、热量与宏量营养素目标、更新时间
2.  全量饮食记录数据：记录唯一标识、记录日期、餐次、原始输入文本、食物明细、热量与营养素汇总数据、创建时间、更新时间
3.  LLM配置数据：模型类型、API密钥、模型版本、接口地址、启用状态、自定义提示词模板、更新时间

### 4.3 数据读写规则
1.  **写入规则**：用户新增、编辑、删除数据后，实时同步写入服务器数据库，写入成功后前端再更新页面展示，确保前后端数据一致；写入失败时，前端给出友好提示，保留用户操作数据，支持重试
2.  **读取规则**：页面初始化时，优先拉取用户基础配置、当日饮食记录；历史数据按日期按需拉取
3.  **一致性要求**：保证前端展示数据与服务器数据库数据完全一致，每次数据操作后，自动刷新相关页面数据

## 五、非功能需求
1.  **性能需求**：页面首屏加载时间≤2s，基础接口平均响应时间≤3s，LLM接口请求响应时间≤8s；支持10000+条饮食记录的存储与查询，查询响应时间≤1s
2.  **易用性需求**：界面极简无冗余元素，核心功能（输入框、营养看板）首页一屏展示，无多余跳转；核心操作步骤不超过2步，无学习成本
3.  **兼容性需求**：适配PC端主流浏览器（Chrome、Edge、Firefox），移动端浏览器自适应，手机端可正常操作所有功能
4.  **可扩展性需求**：架构松耦合，LLM模型可随时新增/替换，数据库可无缝升级，无需改动核心业务代码；预留后续功能扩展接口
5.  **稳定性需求**：服务运行稳定，接口异常时前端给出友好提示，不崩溃、不白屏，不丢失用户已输入内容

## 六、核心业务流程
### 6.1 首次使用流程
部署服务 → 打开应用 → 填写基础身体信息 → 选择减脂/增肌/维持目标 → 系统自动生成营养目标 → 数据自动同步至服务器 → 进入首页开始饮食记录

### 6.2 核心饮食记录流程
首页输入自然语言饮食文本 → 点击提交 → 后端调用LLM解析计算 → 前端展示结构化结果 → 用户可编辑修正 → 确认后数据实时写入服务器 → 看板同步更新摄入进度 → 自动生成当日饮食建议

### 6.3 历史数据查看流程
首页点击日历组件 → 选择目标历史日期 → 前端向服务器发起查询请求 → 服务器返回对应日期的记录与统计数据 → 前端页面展示详情

### 6.4 跨设备同步流程
新设备打开应用 → 自动从服务器拉取全量基础配置与历史记录 → 数据完全同步，可正常操作使用

---

## 七、部署与运维需求
1.  **轻量化部署**：支持一键部署，前端可部署在任意静态站点服务，后端Python服务可部署在本地电脑、轻量云服务器、PaaS平台，支持Docker容器化部署
2.  **零运维要求**：数据库无需单独运维，后端服务启动即可自动完成所有初始化操作；无需定期人工维护数据库
3.  **资源占用要求**：服务运行资源占用极低，支持1核2G配置的轻量服务器稳定运行

---

## 八、基础异常兜底需求
1.  **服务器接口异常兜底**：服务器故障、接口请求失败时，前端给出友好提示，保留用户当前操作的所有数据，支持重试，不丢失用户输入内容
2.  **LLM接口异常兜底**：LLM接口调用失败、超时时，前端给出友好提示，提供手动录入食物、重量、营养数据的入口，保障基础记录功能正常使用
3.  **数据操作异常兜底**：用户编辑、删除数据操作失败时，前端给出明确提示，不修改原有数据，支持用户重试操作
